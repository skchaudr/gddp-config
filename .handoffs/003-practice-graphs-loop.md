## Empirical Reality (2-3 sentences max)

The CLI is proven and the validator is green; zero graphs have been built *with* the new tooling. The blocker was never the schema — it was the friction of hand-rolling nodes, especially inventing encompassing test examples. The shift: stop hand-rolling, generate fast with LLM assist, let the validator be the safety net, then run the loop and build intuition by observing many graphs cheaply.

### Resume point — the loop, in four broad phases

Phase 1 — **Generate, don't hand-roll.** Use `node rapid --llm-draft` (wire a live `GDDP_LLM_*` key) to draft nodes fast; the human edits the last mile. Lower the cost-per-node so making many is cheap.

Phase 2 — **Build 3 throwaway practice graphs**, deliberately varied (tiny greenfield interactive; greenfield via `node import --stdin` = the agent path; brownfield via `--from-graphify graphify-out/graph.json`). Each to `node validate` = 0 errors.

Phase 3 — **Compare against the proven ones.** Run `graph_compare.py` to hold each practice graph against vault-doctor (the gold standard: clean DAG, mechanically-verifiable acceptance) and gddp-runtime. The bar that matters: are acceptance criteria specific enough to convert into a job's `acceptance_ids` and a result's pass/fail keys.

Phase 4 — **Run the loop once for real.** Take one *real* graph, dispatch one `ready` node to Jules → PR → return-router writes a review receipt → you review. Then iterate: observe, learn, adjust generation, repeat. That intuition is the actual deliverable.

Definition of done is unchanged from 002; this is the *how*: make node creation cheap enough to iterate, and use the compare lens to learn what "good" looks like.
